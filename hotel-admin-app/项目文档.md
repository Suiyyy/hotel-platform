# 酒店 O2O 管理平台 — 项目文档

## 一、项目灵感

随着在线旅游行业快速发展，酒店 OTA（在线旅行社）平台已成为用户预订酒店的主要渠道。然而传统的酒店管理后台往往存在以下痛点：

- **信息录入繁琐**：酒店图片需手动粘贴外链、设施服务需手动输入且格式不统一、酒店简介质量参差不齐
- **审核流程割裂**：商家提交后无法实时得知审核进展，管理员也无法第一时间收到新提交通知，双方依赖页面刷新
- **缺乏智能化能力**：用户搜索只能依赖关键词精确匹配，无法理解"帮我找光谷附近安静又便宜的酒店"这类自然语言需求

本项目以"**让酒店管理更智能、更高效**"为核心理念，构建了一套涵盖 C 端用户搜索、B 端商家录入、管理员审核的全链路酒店管理平台，并深度集成 AI 大模型和 WebSocket 实时通信，打造差异化的技术亮点。

## 二、项目亮点

### 1. AI 驱动的智能能力（LangChain + 智谱 GLM）

| 功能 | 说明 |
|------|------|
| **Vibe Search 自然语言搜索** | C 端用户输入自然语言描述，后端调用 GLM-4-Flash 提取结构化搜索意图（关键词、标签、价格、星级、排序），再在数据库中做多维过滤 |
| **AI 润色酒店简介** | B 端商家录入酒店时，一键调用 GLM 将简单描述改写为营销级文案，提升信息质量 |
| **智能降级机制** | GLM 不可用时（超时 10s / 网络异常），自动降级到本地正则规则提取意图，保证功能不中断，前端明确提示降级状态 |

### 2. WebSocket 双向实时通知

| 场景 | 推送方向 | 通知内容 |
|------|----------|----------|
| 商家提交酒店 | → 管理员审核页 | 弹窗提示"商家提交了「xxx」，请及时审核"，列表自动刷新 |
| 管理员审核完成 | → 商家我的酒店页 | 绿色/红色弹窗提示审核通过或拒绝原因，列表自动刷新 |
| 酒店价格变更 | → C 端用户 | 实时推送价格更新 |

### 3. 完整的角色权限体系

- **商家（user）**：录入酒店 → 查看"我的酒店"→ 编辑 → 实时接收审核结果
- **管理员（admin）**：审核管理 → 通过/拒绝（须填原因）→ 上线/下线控制
- 路由级别的权限守卫（ProtectedRoute），未授权自动重定向

### 4. 用户体验细节

- **图片直传**：Ant Design Upload 组件 + 后端 multer，支持拖拽上传、缩略图预览
- **设施标签化**：15 个预设设施选项，勾选替代手动输入，数据格式统一
- **夜间模式**：Ant Design darkAlgorithm 一键切换，localStorage 持久化
- **错误边界**：ErrorBoundary 组件兜底，页面异常时可一键重试
- **响应式布局**：侧边栏自适应折叠，移动端友好

### 4.3 数据流设计

```
用户操作
  ↓

  ↓ 调用

  ↓ 调用

  ↓ fetch + Bearer Token

  ↓ 中间件校验
Redis 读写
  ↓ 结果返回
React 组件更新 UI
```

**WebSocket 实时通信流：**

```
商家提交酒店 → POST /hotels → 后端 saveHotel
                                  ↓
                            broadcastNewHotel()
                                  ↓ WebSocket
                            管理员审核页 onmessage
                                  ↓
                            notification 弹窗 + refreshHotels()

管理员审核 → PATCH /hotels/:id → 后端 saveHotel
                                    ↓
                              broadcastAuditResult()
                                    ↓ WebSocket
                              商家"我的酒店"页 onmessage
                                    ↓
                              notification 弹窗 + loadHotels()
```

### 4.4 认证流程

```
登录请求 → POST /auth/login
             ↓
         验证用户名密码（Redis 查询）
             ↓
         生成 Token（Base64 编码 { userId, username, role }）
             ↓
         返回 { user, token }
             ↓
         前端 setToken → localStorage
             ↓
         后续请求 Authorization: Bearer {token}
             ↓
         authMiddleware 解码验证 → req.user
```

## 五、难点突破

### 难点 1：AI 大模型集成的可靠性问题

**问题**：调用智谱 GLM API 存在网络不稳定、响应超时、返回格式不规范等不确定性，直接影响用户体验。

**解决方案 — 三层防护机制**：

```
第一层：超时控制
  Promise.race([glmCall, timeout(10s)]) → 避免无限等待

第二层：响应解析容错
  用正则 /\{[\s\S]*\}/ 从 GLM 返回中提取 JSON
  即使 GLM 返回了多余文字也能正确解析

第三层：本地降级规则
  当 GLM 完全不可用时，fallbackParseIntent() 用正则提取意图：
  - "500块以内" → maxPrice: 500
  - "五星" → minStar: 5
  - "游泳池" → tags: ["游泳池"]
  前端通过 fallback 标志位提示用户当前为降级模式
```

### 难点 2：WebSocket 与 React 状态同步

**问题**：WebSocket 消息到达时需要更新 React 组件状态和 UI，但 WebSocket 连接生命周期与组件生命周期需要正确对齐，否则会出现内存泄漏或状态丢失。

**解决方案**：

```typescript
// 使用 useRef 持有 WebSocket 实例，useEffect 管理生命周期
const wsRef = useRef<WebSocket | null>(null)

useEffect(() => {
  const ws = new WebSocket(wsUrl)
  wsRef.current = ws

  ws.onmessage = (event) => {
    // 解析消息 → notification 弹窗 → 刷新数据
  }

  return () => { ws.close() }  // 组件卸载时关闭连接
}, [])
```

关键设计：
- `useRef` 避免 WebSocket 实例在重渲染时丢失
- `useEffect` return 清理函数确保组件卸载时断开连接
- 收到消息后调用 `refreshHotels()` / `loadHotels()` 重新拉取数据，保证状态一致性

### 难点 3：跨端口图片资源访问

**问题**：管理端运行在 5173 端口（Vite dev server），后端运行在 3001 端口。上传的图片存储在后端 `uploads/` 目录，图片 URL 为相对路径 `/uploads/xxx.jpg`，浏览器会请求 `localhost:5173/uploads/xxx.jpg` 导致 404。

**解决方案**：

```typescript
// 统一的图片 URL 解析函数
const resolveImageUrl = (url: string) => {
  if (!url) return ''
  // 相对路径拼接后端地址，绝对 URL（如 picsum.photos）原样使用
  return url.startsWith('/') ? `${getBaseUrl()}${url}` : url
}
```

兼容两种图片来源：
- 外链图片（`https://picsum.photos/...`）→ 直接使用
- 上传图片（`/uploads/xxx.jpg`）→ 拼接为 `http://localhost:3001/uploads/xxx.jpg`

### 难点 4：角色感知的路由与 UI 系统

**问题**：同一套管理后台需要同时服务商家和管理员两种角色，路由、侧边栏菜单、操作权限均不同。

**解决方案 — 多层权限控制**：

```
路由层：ProtectedRoute 组件
  ├── 未登录 → 重定向到登录页
  ├── 角色不匹配 → 重定向到首页
  └── 通过 → 渲染目标页面

布局层：AdminLayout 侧边栏
  ├── role === 'user'  → 显示"我的酒店" + "录入酒店"
  └── role === 'admin' → 显示"审核管理"

页面层：按钮/操作级别
  ├── 审核页：仅 pending 状态显示"通过/拒绝"按钮
  ├── 审核页：仅 approved 状态显示"上线/下线"按钮
  └── 我的酒店：仅 isOnline 为 true 时显示"下线"按钮
```

### 难点 5：表单状态与文件上传的协同

**问题**：Ant Design Form 管理表单值，但 Upload 组件的文件状态（fileList）独立于 Form 之外。上传完成后需要将服务端返回的 URL 同步到 Form 的 `imageUrl` 字段，编辑模式下还需要从已有 URL 反向初始化 fileList。

**解决方案**：

```typescript
// 上传完成 → 写入 Form
const handleUploadChange = (info) => {
  setFileList(info.fileList)
  const file = info.fileList[0]
  if (file?.status === 'done' && file.response?.url) {
    form.setFieldsValue({ imageUrl: file.response.url })  // 同步到表单
  }
}

// 编辑模式 → 从 URL 初始化 fileList
useEffect(() => {
  if (hotel?.imageUrl) {
    setFileList([{
      uid: '-1', name: 'hotel-image', status: 'done',
      url: hotel.imageUrl.startsWith('http')
        ? hotel.imageUrl
        : `${getBaseUrl()}${hotel.imageUrl}`
    }])
  }
}, [hotel])
```

Form 中放置一个 `<Input type="hidden">` 承载 `imageUrl` 值，Upload 组件通过 `form.setFieldsValue` 桥接二者。
