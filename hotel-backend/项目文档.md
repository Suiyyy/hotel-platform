# 酒店 O2O 平台后端服务 — 项目文档

## 一、项目灵感

本后端服务是酒店 O2O 预订平台的核心枢纽，同时为 C 端用户 App 和 B 端管理后台提供统一的数据服务。设计初衷是解决以下问题：

- **数据一致性**：多端共享同一套酒店和用户数据，避免数据孤岛
- **实时性需求**：酒店价格变动、审核状态变更需要即时通知各端，传统轮询方案延迟高且浪费资源
- **智能化升级**：传统关键词搜索无法理解用户的模糊需求，需要引入 AI 大模型做自然语言理解
- **文件管理**：酒店图片需要服务端存储和分发，不能依赖外部图床

基于以上需求，后端采用 Express + Redis + WebSocket + LangChain 的技术组合，构建了一套轻量但功能完整的服务架构。

## 二、项目亮点

### 1. AI 服务层（LangChain + 智谱 GLM-4-Flash）

后端集成了两项 AI 能力，均内置三层容错机制：

| 能力 | 接口 | 说明 |
|------|------|------|
| **意图解析** | `POST /ai/search` | 接收自然语言，调用 GLM 提取结构化搜索条件（关键词、标签、价格、星级、排序），再对 Redis 中的酒店数据做多维过滤 |
| **文案润色** | `POST /ai/polish` | 接收酒店简介原文，调用 GLM 改写为营销级文案，提升信息质量 |

**三层容错机制**：
- 第一层：`Promise.race` 实现 10 秒超时控制，避免 GLM 不可达时请求挂起
- 第二层：正则提取 JSON，兼容 GLM 返回多余文本的场景
- 第三层：本地降级规则 `fallbackParseIntent()`，用正则从自然语言中提取价格、星级、设施等关键信息，保证功能不中断

### 2. WebSocket 实时双向推送

基于 `ws` 库实现三种实时推送场景：

| 事件类型 | 触发时机 | 推送内容 |
|----------|----------|----------|
| `NEW_HOTEL` | 商家提交新酒店 | 酒店名称 → 通知管理员审核 |
| `AUDIT_RESULT` | 管理员审核完成 | 审核状态 + 拒绝原因 → 通知商家 |
| `PRICE_UPDATE` | 酒店价格变更 | 新价格 → 通知 C 端用户 |

所有推送通过统一的 `broadcast()` 函数分发给全部连接的客户端。

### 3. Redis 数据层

- **启动时自动 Seed**：从 `data/hotels.json` 加载 10 条预置酒店数据 + 3 个默认用户，首次启动即可使用
- **Pipeline 批量写入**：`setAllHotels()` 使用 Redis Pipeline 原子化替换全量数据，避免中间状态
- **键值设计**：`hotel:{id}` / `user:{username}`，简洁高效

### 4. 文件上传服务

- multer 处理 `multipart/form-data`，限制 5MB
- 文件名采用时间戳防冲突：`{Date.now()}{ext}`
- `express.static` 提供 `/uploads/` 静态文件访问
- 返回相对路径 `/uploads/xxx.jpg`，前端按需拼接完整 URL

### 5. 中间件体系

```
请求 → CORS → JSON 解析 → authMiddleware（认证）→ validateHotel（校验）→ 路由处理 → 错误兜底
```

- **认证中间件**：解析 Bearer Token，将 `{ userId, username, role }` 挂载到 `req.user`
- **校验中间件**：区分 Create（严格必填）和 Patch（宽松可选），返回结构化错误信息
- **错误兜底**：全局 Error Handler 捕获未处理异常，返回统一 500 响应

### 4.3 API 路由设计

#### 认证模块

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| POST | /auth/register | - | 注册（username ≥ 3 字符, password ≥ 6 字符, role: admin/user） |
| POST | /auth/login | - | 登录，返回 `{ user, token }` |

#### 酒店模块

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| GET | /hotels/public | - | 公开查询（仅 approved + online），支持 keyword/sort/order/page/pageSize |
| GET | /hotels | - | 获取全部酒店（管理员用） |
| GET | /hotels/mine | Bearer | 获取当前商家的酒店（按 merchantId 过滤） |
| POST | /hotels | Bearer | 创建酒店，自动设 status=pending, merchantId=当前用户 |
| PATCH | /hotels/:id | - | 更新酒店（含审核状态变更），触发 WebSocket 推送 |
| DELETE | /hotels/:id | - | 软删除（设 isOnline=false） |
| PATCH | /hotels/:id/restore | - | 恢复上线（仅 approved 状态允许） |

#### 文件模块

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /upload | 上传图片（field: file, 限 5MB），返回 `{ url }` |
| GET | /uploads/* | 静态文件访问 |

#### AI 模块

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /ai/search | 自然语言搜索，body: `{ query }` → 返回 `{ data, total, filters, fallback }` |
| POST | /ai/polish | 文案润色，body: `{ text }` → 返回 `{ result, fallback }` |

## 五、难点突破

### 难点 1：GLM 大模型调用的不确定性

**问题**：第三方 AI API 存在网络延迟、超时、返回格式不可控等问题。在生产环境中，任何外部依赖的不可用都不应导致核心功能瘫痪。

**解决方案 — 超时 + 解析容错 + 本地降级**：

```typescript
// 1. 超时控制：Promise.race 竞争
const response = await withTimeout(chatModel.invoke(messages), 10_000)

// 2. 响应解析容错：正则提取 JSON
const jsonMatch = text.match(/\{[\s\S]*\}/)
if (!jsonMatch) return { intent: fallbackParseIntent(input), fallback: true }

// 3. 本地降级：正则规则引擎
function fallbackParseIntent(input: string): ISearchIntent {
  // "500块以内" → maxPrice: 500
  // "五星" → minStar: 5
  // "游泳池" → tags: ["游泳池"]
  // "便宜" → sort: "price", order: "asc"
}
```

三层机制确保：GLM 可用时享受智能体验，不可用时无缝降级到规则匹配，用户始终能获得搜索结果。

### 难点 2：WebSocket 与 HTTP 共存

**问题**：Express 是 HTTP 框架，WebSocket 需要独立的连接升级机制。两者需要共享同一个端口，且 WebSocket 广播需要在 HTTP 路由处理函数中触发。

**解决方案 — 共享 HTTP Server**：

```typescript
// 创建原生 HTTP Server，同时挂载 Express 和 WebSocket
const server = http.createServer(app)
setupWebSocket(server)  // ws 监听同一 server 的 upgrade 事件

// 路由中触发广播（通过模块级变量 wss）
app.post('/hotels', async (req, res) => {
  await saveHotel(newHotel)
  broadcastNewHotel(newHotel)  // WebSocket 广播
  res.status(201).json(newHotel)
})
```

`ws` 库通过监听 HTTP Server 的 `upgrade` 事件自动处理协议升级，HTTP 和 WebSocket 共用 3001 端口。广播函数通过模块闭包访问 `wss` 实例，路由处理函数无需关心 WebSocket 细节。

### 难点 3：Redis 数据初始化的幂等性

**问题**：后端启动时需要从 JSON 文件 Seed 初始数据到 Redis。但重复启动不应覆盖运行中产生的新数据（如商家新提交的酒店）。

**解决方案 — 存在性检查 + Pipeline 批量写入**：

```typescript
async function initRedis() {
  await redis.connect()

  // 仅当 Redis 中无酒店数据时才 Seed
  const existingKeys = await redis.keys('hotel:*')
  if (existingKeys.length === 0) {
    const hotels = JSON.parse(fs.readFileSync('data/hotels.json'))
    const pipeline = redis.pipeline()
    hotels.forEach(h => pipeline.set(`hotel:${h.id}`, JSON.stringify(h)))
    await pipeline.exec()
  }

  // 用户同理：仅当不存在时创建
  for (const user of defaultUsers) {
    const existing = await redis.get(`user:${user.username}`)
    if (!existing) await redis.set(`user:${user.username}`, JSON.stringify(user))
  }
}
```

首次启动自动 Seed，后续重启保留已有数据。Pipeline 保证批量写入的原子性和性能。

### 难点 4：文件上传与跨域访问

**问题**：管理端（5173）上传图片到后端（3001），存在跨域问题。且上传后的图片 URL 需要被管理端和用户端都能正确访问。

**解决方案**：

```typescript
// 1. CORS 全局放开（开发环境）
app.use(cors())

// 2. multer 磁盘存储 + 时间戳文件名
const storage = multer.diskStorage({
  destination: path.resolve(process.cwd(), 'uploads'),
  filename: (_req, file, cb) => cb(null, `${Date.now()}${path.extname(file.originalname)}`)
})

// 3. 静态文件服务
app.use('/uploads', express.static(path.resolve(process.cwd(), 'uploads')))

// 4. 返回相对路径，前端按需拼接
res.json({ url: `/uploads/${req.file.filename}` })
```

后端只返回相对路径 `/uploads/xxx.jpg`，前端根据运行环境拼接完整 URL。这种设计解耦了前后端的部署地址，便于后续迁移到 CDN 或对象存储。

### 难点 5：认证与权限的分层设计

**问题**：部分接口需要认证（如创建酒店、查看我的酒店），部分接口公开访问（如公开酒店列表），还需要区分管理员和商家的操作权限。

**解决方案 — 中间件组合模式**：

```typescript
// 公开接口：无中间件
app.get('/hotels/public', handler)

// 需认证：authMiddleware
app.post('/hotels', authMiddleware, validateHotelCreate, handler)
app.get('/hotels/mine', authMiddleware, handler)

// 需特定角色：requireRole
app.get('/admin/stats', authMiddleware, requireRole('admin'), handler)
```

`authMiddleware` 负责 Token 解析和用户身份挂载，`requireRole` 在此基础上检查角色。两个中间件可自由组合，路由级别声明权限需求，职责清晰。
